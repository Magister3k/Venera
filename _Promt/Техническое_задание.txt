1. Функционал программы:
1.1. Два режима работы (параметр в файле конфигурации): в качестве приложения в системном трее (библиотека getlantern/systray) и службы Windows.
1.2. Входные данные в отдельном процессе обрабатываются программой Tshark, результаты помещаются в базу СУБД PostgreSQL с промежуточной обработкой в СУБД DragonflyDB.
1.3. Входные данные могут поступать непрерывным потоком с сетевой карты, из папки с файлами или отдельного файла.
1.4. Процессы обработки могут независимо создаваться, запускаться, останавливаться и удаляться.
1.5. Одновременно может быть запущено до 20 рабочих процессов (параметр в файле конфигурации).
1.6. Настройка программы и управление процессами обработки осуществляется с рабочих мест по сети через веб-интерфейс.
1.7. Хранение списка фильтрации данных в текстовом файле generic.flt (путь в файле конфигурации) формата:
символ "+" _ключ_|_название_
символ "-" _значение_
символ "-" _значение_
..,
где "+" - элементы белого списка ключей (1-й уровень), "-" - элементы черного списка значений (2-ой уровень).
1.8. Хранение списка значений на контроле в текстовом файле generic.ctr (путь в файле конфигурации) формата csv (_ключ_|_значение_).
1.9. Хранение параметров текущих процессов в текстовом файле processes.toml формата toml с группировкой по ID процесса (тип источника; название источника; для сетевой карты: IP-адрес и UDP-порт / для папки с файлами: путь к папке, режим сканирования подпапок, режим мониторинга новых файлов).
1.10. Хранение алертов в текстовом файле generic.alr формата cel.
1.11. Хранение параметров конфигурации в текстовом файле config.toml формата toml с группировкой по разделам:
- Generic (режим работы, автоматический старт процессов при запуске приложения, максимальное количество процессов, номер порта веб-сервера)
- Paths (путь к файлу podman.exe, путь к файлу tshark.exe, путь к файлу списка фильтрации данных generic.flt, путь к файлу списка значений на контроле generic.ctr, путь к файлу алертов generic.alr, путь к локальному файлу с образом СУБД DragonflyDB, путь к папке для размещения резервных данных СУБД DragonflyDB)
- DragonflyDB (параметры подключения к СУБД DragonflyDB, количество записей для обработки в структуре list СУБД DragonflyDB, время ожидания для обработки записей в структуре list СУБД DragonflyDB)
- PostgreSQL (параметры подключения к СУБД PostgreSQL)
1.12. Динамический веб-интерфейс.
1.13. Вывод в веб-интерфейсе статистики работы в реальном времени.
1.14. Работа в веб-интерфейсе с базой СУБД PostgreSQL.

4. Логика бизнес-процесса:
4.1. Каждый входной процесс ПВ1...ПВn получает данные, заносит их в свою структуру list ЛСТ1...ЛСТn кэш-базы БД1 и увеличивает соответствующий счётчик записей СЗ1...СЗn. Когда значение счётчика записей СЗn достигает установленного порога ПЗ, отправляется сигнал в канал КАН1...КАНn, сигнализируя процессу управления задачами о том, что порог достигнут. Значение счётчика записей СЗn сбрасывается, процесс ПВn продолжает работу.
4.2. Процесс управления задачами следит за каналами КАН1...КАНn и таймерами работы процессов ТП1...ТПn: при получении сигнала в канале КАНn или срабатывании таймера ТПn запускает новый процесс обработки данных ПОn отдельно для каждой структуры ЛСТn где достигнут порог или сработал таймер. С одной структурой может работать только один процесс обработки данных одномоментно, но одновременно может быть запущено до n таких процессов для разных структур. После запуска обработчика увеличиваетcя количество ожидаемых работ.
4.3. Процессы обработки данных ПО1...ПОn работают параллельно и выполняют задачи фильтрации и переноса данных из структуры list ЛСТ1...ЛСТn через структуры sorted set СС1...CCn кэш-базы БД1 в итоговую базу данных БД2. Когда обработка закончена, соответствующая работа помечается как выполненная.

5. Схема работы бизнес-процесса (для каждого из запущенных процессов):
5.1. Получение программой Tshark одного из вариантов входных данных (непрерывный поток с сетевой карты по выбранному IP-адресу и UDP-порту / файлы из указанной папки / отдельный файл) с экспортом в формате json.
5.2. Разбиение потока json на пары ключ-значение с учетом допустимости использования в значениях символа ":" (пул парсеров из библиотеки fastjson).
5.3. Добавление к паре ключ-значение времени фиксации (приведение к формату ключ:значение:время).
5.4. Помещение параметра ключ:значение:время в структуру list СУБД DragonflyDB.
5.5. Каждые N записей в структуре list СУБД DragonflyDB (параметр в файле конфигурации) или период времени T (параметр в файле конфигурации), смотря что наступит раньше, а также при остановке процесса запускается асинхронный процесс их обработки со следующими задачами:
5.5.1. Получение записей из структуры list СУБД DragonflyDB с удалением.
5.5.2. Разделение записей на ключ, значение и время.
5.5.3. Фильтрация ключей согласно белому списку и значений согласно черному списку (из п.1.7 и п.2.2).
5.5.4. Помещение в структуру sorted sets СУБД DragonflyDB пар ключ-значение и времени.
5.5.5. Перемещение всех записей из структуры sorted sets СУБД DragonflyDB в СУБД PostgreSQL в пакетном режиме (библиотека pgx v5).
5.5.6. Закрытие процесса обработки.
